AuthGuard¶

function AuthGuard({ children }: { children: React.ReactNode }) {
  const { user } = useApp();
  if (!user) return <Navigate to="/login" replace />;
  return <>{children}</>;
}

AuthGuard wraps the entire /requester route subtree. Any navigation to a protected route while unauthenticated redirects to /login. After login, the user is sent to /requester/create.

Route Structure¶

<Routes>
  <Route path="/login"       element={user ? <Navigate to="/requester/create" /> : <LoginPage />} />
  <Route path="/requester"   element={<AuthGuard><RequesterLayout /></AuthGuard>}>
    <Route index             element={<Navigate to="create" />} />
    <Route path="create"     element={<CreateRequestPage />} />
    <Route path="material"   element={<MaterialSelectionPage />} />
    <Route path="checkout"   element={<CheckoutPage />} />
    <Route path="container"  element={<ContainerSelectionPage />} />
    <Route path="container-checkout" element={<ContainerCheckoutPage />} />
    <Route path="return-trolley"     element={<ReturnTrolleyPage />} />
    <Route path="requests"   element={<MyRequestsPage />} />
    <Route path="staging"    element={<StagingAreaPage />} />
    <Route path="inventory"  element={<InventoryPage />} />
  </Route>
  <Route path="*"            element={<Navigate to="/login" />} />
</Routes>

The /requester route renders RequesterLayout as its element — this is the shell that provides the header and bottom navigation. All child routes render into the <Outlet /> inside RequesterLayout.

RequesterLayout.tsx¶

<Box sx={{ display: 'flex', flexDirection: 'column', height: '100vh', overflow: 'hidden' }}>
  <TabletHeader />
  <Box sx={{
    mt: '52px',    // offset for fixed header
    mb: '60px',    // offset for fixed bottom nav
    flex: 1,
    minHeight: 0,  // CRITICAL: allows flex child to shrink for scrolling
    overflow: 'hidden',
    display: 'flex',
    flexDirection: 'column',
    bgcolor: 'background.default',
  }}>
    <Outlet />   // ← page renders here
  </Box>
  <BottomNav />
</Box>

The minHeight: 0 Pattern¶

This is the most important layout rule in the application. Every flex container in the scroll chain must have minHeight: 0 — otherwise a flex child with overflow: auto cannot shrink below its content height, preventing scrollbars from working and pushing fixed footer bars off-screen.

The required pattern for every page:

{/* Outer page box — fills the layout outlet */}
<Box sx={{ display: 'flex', flexDirection: 'column', flex: 1, minHeight: 0, overflow: 'hidden' }}>

  {/* Fixed sub-header — never scrolls */}
  <Box sx={{ ..., flexShrink: 0 }}>
    <IconButton>Back</IconButton>
    <Typography>Page Title</Typography>
  </Box>

  {/* Scrollable content area */}
  <Box sx={{ flex: 1, minHeight: 0, overflow: 'auto', p: 2.5 }}>
    {/* page content */}
  </Box>

  {/* Fixed submit bar — never scrolls */}
  <Box sx={{ p: 2, borderTop: '1px solid', borderColor: 'divider', flexShrink: 0 }}>
    <Button>Submit</Button>
  </Box>

</Box>

Omitting minHeight: 0 on either the outer box or the scrollable content box will cause layout overflow on the 528px viewport.

Component Architecture¶

TabletHeader.tsx¶

  • position: fixed MUI AppBar at zIndex: 1200
  • Height: 52px (forced via minHeight: '52px !important' on Toolbar)
  • Contains: brand logo, station name, workflow selector dropdown, connection status chip, operator avatar
  • The workflow selector uses MUI Select directly in the header — changing it calls setActiveWorkflow() on AppContext, instantly updating the active workflow across all pages
  • Station name has maxWidth: 180, overflow: 'hidden', textOverflow: 'ellipsis' to prevent overflow on long names
  • Renders null if !user || !activeWorkflow (during login flow)

BottomNav.tsx¶

  • position: fixed MUI Paper at zIndex: 1100
  • Height: 60px
  • 4 navigation items:
  • Create Request (/requester/create) — shows cart count Badge when cart.length > 0
  • My Requests (/requester/requests)
  • Staging Area (/requester/staging)
  • Inventory (/requester/inventory)
  • Active tab detected via pathname.startsWith(item.path) — correctly highlights the parent tab even on sub-routes like /requester/material (which highlights "Create Request")

Routing Details¶

Navigation Patterns¶

Three navigation patterns are used:

  1. useNavigate(path) — direct programmatic navigation (after form submit, button click)
  2. useNavigate(-1) — browser history back (Back arrow buttons in sub-headers)
  3. <Navigate to="..." replace /> — redirect without history entry (auth guard, post-login)

Sub-route Flow (not using nested routes)¶

Some pages have multi-step flows implemented within the same route using local useState:

  • ReturnTrolleyPage: method = null | 'qr' | 'details' controls which view renders
  • StagingAreaPage: selectedArea = null | StagingArea controls list vs. detail view

This keeps the URL stable (no back/forward navigation between sub-steps), which is appropriate for touch-device industrial UIs where back-button confusion must be minimised.

Data Layer¶

Currently all data is mocked in src/data/mock.ts. The mock layer exports:

Export Type Description
mockWorkflows Workflow[] 3 workflows with different strategies
mockDeviceUser DeviceUser Single operator with all 3 workflows
mockMaterials MaterialSKU[] 4 SKUs with 9 Sub-SKU types total
mockContainers Container[] 3 container types (Trolley/Pallet/Bin) with 7 subtypes
mockStagingAreas StagingArea[] 2 areas (4×10 and 3×8 grid)
mockRequests Request[] 4 requests across different workflows/statuses
mockInventory InventoryRow[] 8 rows of inventory data

In production, these imports would be replaced with API calls. The function signatures of the context actions (addToCart, setContainerCart, etc.) are designed to remain unchanged.
