My Requests Page¶

Route: /requester/requests Component: src/pages/requester/MyRequestsPage.tsx Auth: Protected Bottom Nav: "My Requests" tab (index 1)

Purpose¶

Shows the operator a filterable list of all requests scoped to the active workflow. Provides status tracking for submitted material, container, and return trolley requests.

UI Components¶

Header¶

  • Title "My Requests" (fontWeight: 700)
  • Caption: {activeWorkflow?.name} · {workflowRequests.length} request(s)
  • Count reflects total workflow requests (not filtered count)

Filter Tabs¶

5 filter tabs:

Index Label Statuses shown
0 All All
1 Pending pending
2 In Progress in_progress, awaiting_confirmation
3 Completed completed
4 Failed failed, breakdown

MUI Tabs with minHeight: 42 (slightly smaller than the 44px default for density). Indicator height: 2px.

Request Card¶

┌──────────────────────────────────────────┐
│ req-001           [Material]    [In Progress] │
│ Widget A – T1 × 2                 10:32 AM │
├──────────────────────────────────────────┤
│ Workflow: Assembly Line A    ● AMR En Route │

Main row (top): - Left: Request ID in fontWeight: 700, type Chip (grey #F0F3F5 background) - Right: Status Chip (colour-coded) + timestamp caption

Footer row (bottom): - Workflow name in grey text - "● AMR En Route" animated indicator — only shown when status === 'in_progress' - Green 6px circle with CSS pulse animation (same keyframes as other status indicators) - Typography in teal with fontWeight: 600

UX Logic¶

Workflow Filtering¶

// First filter: only show requests for the active workflow
const workflowRequests = mockRequests.filter((r) => r.workflowId === activeWorkflow?.id);

// Then apply status tab filter
const filtered = workflowRequests.filter((r) => {
  if (tabIdx === 0) return true;
  return (
    (tabIdx === 1 && r.status === 'pending') ||
    (tabIdx === 2 && (r.status === 'in_progress' || r.status === 'awaiting_confirmation')) ||
    (tabIdx === 3 && r.status === 'completed') ||
    (tabIdx === 4 && (r.status === 'failed' || r.status === 'breakdown'))
  );
});

Switching workflows in the header (via TabletHeader dropdown) will change activeWorkflow.id, causing this component to re-render and show only the new workflow's requests.

AMR En Route Indicator¶

The animated dot is a purely visual hint — it has no polling or real-time update in the prototype. In production this would connect to a WebSocket or polling endpoint.

Edge Cases¶

Scenario Behaviour
Switch to workflow with no requests Empty state shown
Active workflow = QC Station B Only req-003 appears (different workflowId)
All requests completed Only Completed tab has entries
breakdown status Shows in "Failed" tab
awaiting_confirmation status Shows in "In Progress" tab
