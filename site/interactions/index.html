Interactions & Navigation¶

Navigation Architecture¶

Primary Navigation (BottomNav)¶

The BottomNav is always visible once logged in. It provides access to the 4 main sections:

[+ Create Request]  [üìã My Requests]  [üóì Staging Area]  [üì¶ Inventory]
   /requester/create   /requester/requests  /requester/staging  /requester/inventory

Active tab detection uses pathname.startsWith(item.path). This means: - Navigating to /requester/material highlights "Create Request" - Navigating to /requester/checkout highlights "Create Request" - Navigating to /requester/return-trolley highlights "Create Request"

Cart badge: When cart.length > 0, the Create Request tab icon shows a teal Badge with the item count.

Sub-page Navigation (Sub-headers)¶

Pages that are reached via a flow (not directly from BottomNav) have a fixed sub-header with a back arrow (ArrowBackIcon). These use:

// History back (from a specific previous page)
<IconButton onClick={() => navigate(-1)}>

// Internal state back (no history change)
<IconButton onClick={() => setMethod(null)}>

Workflow Context (TabletHeader)¶

The workflow selector in the header is always accessible. Changing it: 1. Updates activeWorkflow in context 2. Immediately re-renders all pages that read activeWorkflow 3. MyRequestsPage filter changes to show the new workflow's requests

This is designed so supervisors or multi-workflow stations can switch context without logging out.

Interaction Patterns¶

Quantity Steppers¶

Used in: MaterialSelectionPage (grid + list), CheckoutPage (cart items + inline containers), ContainerSelectionPage, ReturnTrolleyPage (does not have one).

Standard pattern: 

[‚àí] value [+]
- ‚àí disabled at qty <= 1 (or at minimum allowed value) - + disabled at qty >= maxQty (or when unavailable) - borderRadius: 0 on both buttons for seamless inline appearance - The value display is not an input ‚Äî it is Typography only. There is no direct text entry for quantities.

Confirm Dialogs¶

All destructive or irreversible actions (submit request, submit container order, confirm return pickup) use a confirmation dialog pattern:

[Dialog title]
[Summary of what will happen + key data]

[Cancel]    [Confirm & Submit]
  • maxWidth="xs" fullWidth ‚Äî 480px max on desktop, full-width on small screens
  • Cancel is left, confirm is right (standard MUI DialogActions order)
  • borderRadius: 10 on dialog paper (from theme override)

Success Screens¶

All final submissions show a full-page success state (no navigation change ‚Äî same route):

          ‚óè (coloured circle)
          Action Completed!

          [descriptive message]

    [View My Requests]    [New Request / Home]
  • Left button: secondary action (view history)
  • Right button: primary action (start fresh)

Success screen circle colours: - Material checkout: Teal rgba(0,150,136,0.1) with teal CheckCircle - Container checkout: Blue rgba(21,101,192,0.1) with blue CheckCircle - Return trolley: Purple rgba(106,27,154,0.1) with purple CheckCircle

Loading/Async States¶

Only one async interaction exists in the prototype: QR scan simulation (1.2 second delay).

Pattern: - Button disabled (disabled={qrScanning}) - Button text changes ("Scanning‚Ä¶") - Custom disabled style: '&.Mui-disabled': { bgcolor: 'rgba(106,27,154,0.5)', color: '#fff' } - Animated pulse indicator below button during scan

Collapse Sections¶

CheckoutPage uses MUI Collapse for the "Add Containers" section. The header row is fully tappable (not just an icon) and shows a summary preview when collapsed with data.

Accessibility Notes¶

Touch Targets¶

All primary interactive elements meet the 44px minimum (enforced via MUI theme). Some inline stepper buttons use 36px ‚Äî acceptable in context where they are paired with other controls.

Keyboard Support¶

  • Enter key in LoginPage submits the form
  • All MUI interactive elements support keyboard navigation (Tab, Enter, Space, Arrow keys)
  • Dialogs trap focus correctly (MUI default behaviour)

Colour Contrast¶

  • Body text #1A2332 on #F5F7F9 background ‚Äî high contrast
  • Status chips use colour + text label (not colour alone) for accessibility
  • Icons are always accompanied by text labels in important contexts

Screen Reader¶

  • MUI components use ARIA attributes by default
  • aria-label should be added to icon-only buttons in production (currently not set on back arrows, delete icons, etc.)
