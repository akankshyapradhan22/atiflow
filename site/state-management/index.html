State Management¶

All global state lives in a single React Context: AppContext (src/context/AppContext.tsx). It is provided at the root of the application by AppProvider wrapping the entire router.

Access it in any component with:

import { useApp } from '../../context/AppContext';
const { user, cart, addToCart } = useApp();

State Shape¶

interface AppState {
  user: DeviceUser | null;          // null = not logged in
  activeWorkflow: Workflow | null;  // null = not logged in
  cart: CartItem[];                 // material items selected for checkout
  containerCart: ContainerCartItem[]; // container items (separate flow)
  returnTrolleyEnabled: boolean;    // toggle on CheckoutPage
}

State Variables¶

user: DeviceUser | null¶

The currently authenticated device operator.

  • Initial value: null (unauthenticated)
  • Set by: login() action (sets to mockDeviceUser with the entered username applied)
  • Cleared by: logout()
  • Used by: AuthGuard, TabletHeader, MyRequestsPage (workflow filtering)

The user.workflows array is the source of truth for which workflows are available in the header dropdown.

activeWorkflow: Workflow | null¶

The currently selected workflow. All material requests, container orders, and return trolleys are scoped to this workflow.

  • Initial value: null
  • Set by: login() (auto-selects first workflow), setActiveWorkflow() (via header dropdown)
  • Cleared by: logout()
  • Used by: TabletHeader (dropdown), CreateRequestPage (info strip), MaterialSelectionPage (header caption), CheckoutPage (summary, confirm dialog, return trolley chip text), ContainerCheckoutPage (header caption, confirm dialog), ReturnTrolleyPage (confirm dialog, DetailCard), MyRequestsPage (workflow-scoped request filter)

Workflow Properties¶

interface Workflow {
  id: string;
  name: string;
  assignmentStrategy: 'request-based' | 'on-route';
  confirmationMode: 'auto' | 'manual';
}
assignmentStrategy Effect
'request-based' AMR trip created immediately on submission. Success message: "Trip will be fulfilled immediately by an AMR." Return trolley chip: "Trip scheduled immediately on delivery."
'on-route' AMR trip created only when the AMR scans the trolley QR code (FIFO queue). Success message: "Trip will be assigned when the AMR scans the trolley QR code." Return trolley chip: "AMR will collect on QR scan (FIFO)."
confirmationMode Effect
'auto' No dispatcher needed. CreateRequestPage info strip shows "Not required."
'manual' Dispatcher must confirm the request. CreateRequestPage info strip shows "Required" in amber (#E65100).

cart: CartItem[]¶

The material items staged for submission.

  • Initial value: []
  • Modified by: addToCart(), updateCartQty(), removeFromCart(), clearCart()
  • Used by: MaterialSelectionPage (in-cart state), CheckoutPage (review list), BottomNav (cart badge count), CreateRequestPage (cart banner + Checkout button)
interface CartItem {
  subSkuTypeId: string;     // unique key
  subSkuTypeName: string;   // display name "Widget A – Sub-SKU Type 1"
  skuName: string;          // parent SKU name
  quantity: number;         // current selected quantity
  maxQty: number;           // max allowed quantity (from SubSKUType)
}

Cart Actions¶

Action Signature Behaviour
addToCart (item: CartItem) => void If item with same subSkuTypeId already exists, increments quantity (capped at maxQty). Otherwise appends.
updateCartQty (subSkuTypeId: string, qty: number) => void Replaces quantity directly — used by CheckoutPage quantity stepper.
removeFromCart (subSkuTypeId: string) => void Removes item from cart.
clearCart () => void Empties cart entirely — called after successful submission.

containerCart: ContainerCartItem[]¶

Container items selected in ContainerSelectionPage. Separate from cart because containers go through a different checkout flow.

  • Initial value: []
  • Set by: setContainerCart(items) — called at once with the full selection array
  • Cleared by: clearContainerCart() — called after successful submission
  • Used by: ContainerCheckoutPage
interface ContainerCartItem {
  containerId: string;        // container type ID
  containerType: ContainerType; // 'trolley' | 'pallet' | 'bin'
  subtypeId: string;          // subtype ID
  subtypeName: string;        // display name
  quantity: number;
}

returnTrolleyEnabled: boolean¶

Controls whether an empty trolley return is automatically scheduled alongside the material request.

  • Initial value: true (on by default)
  • Set by: setReturnTrolley(val: boolean) — controlled by the Switch in CheckoutPage
  • Persisted across sessions: No — resets on logout (currently resets on login only via context re-initialisation)
  • Used by: CheckoutPage (toggle UI + return trolley section visibility + success message text)

Actions Reference¶

interface AppContextValue extends AppState {
  login: (username: string, password: string) => boolean;
  logout: () => void;
  setActiveWorkflow: (w: Workflow) => void;
  addToCart: (item: CartItem) => void;
  updateCartQty: (subSkuTypeId: string, qty: number) => void;
  removeFromCart: (subSkuTypeId: string) => void;
  clearCart: () => void;
  setContainerCart: (items: ContainerCartItem[]) => void;
  clearContainerCart: () => void;
  setReturnTrolley: (val: boolean) => void;
}

login(username, password): boolean¶

Mock login — accepts any non-empty string pair.

const login = (username: string, password: string): boolean => {
  if (username.trim() && password.trim()) {
    setUser({ ...mockDeviceUser, username });
    setActiveWorkflowState(mockDeviceUser.workflows[0] ?? null);
    return true;
  }
  return false;
};
  • Returns true on success (caller navigates to /requester/create)
  • Returns false on failure (caller shows error alert)
  • The username field is overridden with the entered value so the header displays the actual input

logout()¶

const logout = () => {
  setUser(null);
  setActiveWorkflowState(null);
  setCart([]);
  setContainerCartState([]);
};

Clears all user state. Does not reset returnTrolleyEnabled (it will be reset next time context re-initialises on next login via useState(true) initial value).

State Flow Diagram¶

Login
  └─> login() ──> user set, activeWorkflow set ──> navigate('/requester/create')

CreateRequestPage
  ├─> navigate('/requester/material') ──> MaterialSelectionPage
  │       └─> addToCart() / removeFromCart()
  │               └─> navigate('/requester/checkout') ──> CheckoutPage
  │                       ├─> updateCartQty() / removeFromCart()
  │                       ├─> setReturnTrolley()
  │                       ├─> [inline container add – local state only]
  │                       └─> Submit ──> clearCart() ──> navigate('/requester/create')
  │
  └─> navigate('/requester/return-trolley') ──> ReturnTrolleyPage
          └─> [all state local – no context writes]

ContainerSelectionPage
  └─> setContainerCart(items) ──> navigate('/requester/container-checkout')
          └─> ContainerCheckoutPage
                  └─> Submit ──> clearContainerCart() ──> navigate('/requester/create')

TabletHeader
  └─> setActiveWorkflow(wf) ──> ALL pages re-render with new workflow context

Local State vs. Context State¶

Not all state lives in context. The following is local to each page (React useState):

Page Local State
LoginPage stationCode, password, error
MaterialSelectionPage search, quantities (pending input before add), viewMode
CheckoutPage confirmOpen, submitted, containerExpanded, addedContainers, selContainerIdx, selSubtypeId, containerQty
ContainerSelectionPage tabIdx, quantities, selected (Set of subtypeIds)
ContainerCheckoutPage confirmOpen, submitted
ReturnTrolleyPage method, resolved, confirmOpen, submitted, qrScanning, containerId, containerTypeIdx, subtypeId, notFound
MyRequestsPage tabIdx (filter tab index)
StagingAreaPage selectedArea, displayMode, viewLayout, edits, editingCell, savedCount
InventoryPage search
